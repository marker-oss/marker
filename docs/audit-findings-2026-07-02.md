## 1. P0 — деньги и активация

### P0-1. WB: рекламный тракт — два латентных бага (сработают при реальных кампаниях)
Уточнение к «WB April ad_cost = 0.25 ₽»: расследование 2026-05-04 по raw-ответам WB API
уже установило, что апрельские копейки **реальны** (спящие авто-промо `days[].sum=0.05…`,
платных кампаний не было) — это не баг синка. Но аудит нашёл два дефекта кода, которые
сделают WB-рекламу неверной, как только продавец запустит бюджетные кампании:

- **(a) Фильтр статусов кампаний отбрасывает завершённые.** `ingest.clj:146` →
  `wb/api.clj:134-135`: `ad-campaigns` по умолчанию фильтрует `#{9 11}` (active/paused).
  Кампании со статусом 7 (завершена) исключаются → ре-синк прошлого периода после
  завершения кампаний теряет их спенд целиком (история рекламы невоспроизводима).
  Согласуется с raw-батчами `wb/ad_stats` 15.06/26.06 с `item_count = 0`.
- **(b) Аллокатор джойнит дневной спенд к недельным датам.** `materialize.clj:1043-1052`
  ключует finance-строки по `substr(date_from,1,10)` — а у WB это начало *недельного*
  отчёта, не день события (`event_date`). Совпадение только если день рекламы == старт
  недели. Измерено на живых данных: аллоцировано 0.22 из 4.95 ₽ (**4.4 %**) — тот же
  дефект, что отложенная заметка «canonical path populates only ~7% of rows» от 04.05.
  Спенд по nm_id без строки на точную дату теряется молча, даже без инкремента
  счётчика unallocated (`materialize.clj:1127-1131`).
- Направление фикса: (a) включить статус 7 при ingest прошлых периодов;
  (b) джойнить по `COALESCE(event_date, substr(date_from,1,10))`.
- Следствие: когда кампании станут реальными, стек ДРР/max-ДРР/headroom для WB
  (формулы теперь корректны — 014) молча потеряет ~95 % спенда и историю завершённых
  кампаний. Это конкретизирует «013-B unwritten» из coverage-аудита.

### P0-3. Ozon Performance (011): латентный ×30–90 множитель ad_cost — чинить ДО включения кредов
`ad_campaign_stats` пуст (0 строк), `finance.ad_cost` по Ozon всюду 0 — обвязка 011 ещё
ни разу не отработала на живых данных. При активации выстрелят четыре бага:

- **(a) Спенд артикула SET-ится на каждую строку.** `materialize.clj:1457-1464`:
  `UPDATE finance SET ad_cost = <полный спенд артикула за период> WHERE article = ? AND
  <окно>` — выполняется ПОСЛЕ respread, когда у артикула десятки дневных строк.
  `SUM(ad_cost)` (то, что читает P&L через `db/ad-spend-by-article`) завышается в
  N-строк раз (в апрельских данных 60–90 строк на топ-артикул). Реконсиляция FR-008
  поймает, но только залогирует (`materialize.clj:1493-1499`). Тот же дефект в
  `canonical/ad/materialize.clj:76-85`.
- **(b) Async-статистика без campaign_id размножается на все кампании.** Отчёт
  запрашивается по всем кампаниям сразу, строки не несут id кампании
  (`performance/transform.clj:85-97`), а `build-ozon-ad-campaign-stats`
  (`materialize.clj:1353-1367`) маппит полный вектор строк на каждую кампанию →
  в `ad_campaign_stats` K× реальных сумм, у всех кампаний одинаковые цифры.
- **(c) Async-строки штампуются id первой кампании** (`materialize.clj:1410-1416`) →
  displacement-merge по (campaign, article, day) не вытесняет базовые строки кампаний
  2..K → двойной счёт спенда в `finance.ad_cost` при >1 кампании и `:fetch-stats?`.
- **(d) PK ad_spend схлопывает мульти-SKU артикулы** (`db.clj:874-887` PK без sku,
  а атрибуция эмитит по-SKU) → INSERT OR REPLACE теряет строки.
- **(e) Побочно:** `materialize-ozon-ad-cost!` перезаписывает `event_date_source`
  атрибуцией рекламы (`materialize.clj:1459-1463`) — `flat`-строки перепомечаются как
  `api`, портя date-basis аудит; и обнуляет transaction-derived promo ad-cost
  (`MarketplaceMarketingActionCostOperation`) при наличии `:ad_performance` raw.

## 2. P1 — высокие

### Расчётный слой
- **Знак `:additional` в Pulse-расходах.** `marker.clj:293-300` (`sum-other-costs`)
  суммирует `:additional` (доплаты — кредит продавцу; в P&L он *прибавляется* к
  gross-profit, `pnl.clj:248`) как положительный расход → «Прочее» и «Итого расходы»
  завышены на 2×additional относительно подписанной истины; блок расходов перестаёт
  сходиться с revenue − profit. ПОДТВЕРЖДЕНО.
- **`profit-per-sale` меняет знак при отрицательном net-qty.** `math/safe-div`
  (`util/math.clj:26-31`) охраняет только ноль; канон P&L.5 обещает 0 при net-qty ≤ 0.
  Убыток −5000 ₽ при 10 продажах / 20 возвратах → «+500 ₽ на продажу». UE-тоталы этот
  же баг уже чинили клампом (`unit_economics.clj:198`) — P&L разошёлся с UE. ПОДТВЕРЖДЕНО.
- **GMROI-кластер** (016 FR-009/010, коммит b3a2f76):
  числитель = `for-pay − cogs − ads` вместо net-profit (без логистики/хранения/штрафов;
  `marker.clj:1103-1107, 1429-1430`); числитель не ограничен covered-days при
  знаменателе, усреднённом только по покрытым дням (→ ~2× завышение при 50 % покрытия);
  на `/stocks/overview` GMROI считается с `:history []` → всегда `:not-applicable`
  (метрика структурно мертва на этом эндпоинте, `marker.clj:1112`).
- **YM sales-таблица осталась на старом базисе.** 012 починил finance-путь
  (gross = BUYER + subsidy), но `->sale-row` в `ym/transform.clj:159-174,191` всё ещё
  пишет `for-pay = MARKETPLACE.total − bidFee` (bidFee — это bid-*cap*, в ~351× больше
  реального списания; сам файл это документирует на :415-420). `domain/sales.clj:81`
  суммирует `sales.for_pay` как `:revenue` → все sales-based поверхности YM (Pulse-стиль)
  на неверном базисе. Это и есть «hardening TODO» из memory.

### API / безопасность
- **DoS: бесконечный цикл на неаутентифицированном GET.**
  `treasury/cashflow.clj:67-80` (`months-in-range`): при `from > to` курсор доходит до
  `"10000-01"`, парсится как год 1000/месяц 0 и зацикливается навсегда, `acc` растёт
  без границы. `GET /api/v1/treasury/cashflow?from=2026-05-01&to=2026-01-31` (одна
  опечатка в date-picker) вешает Jetty-поток на 100 % CPU до OOM. GET-ы не требуют
  API-ключа. Отсутствующие from/to → NPE-500. Арифметика цикла перепроверена вручную.
- **GET /api/v1/opex пишет в БД с невалидированным period.** `tax_opex.clj:113-126`
  вызывает `materialize-rules!` (INSERT) на GET; period проверяется только на nil, не на
  `YYYY-MM` → неаутентифицированный GET создаёт opex_rows с мусорным period_month
  (сироты, не попадающие ни в одно окно чтения). POST-путь при этом Malli-валидирован.
- **Auth-модель:** `wrap-api-key` гейтит только POST/PUT/DELETE под `/api`; все GET
  открыты на уровне приложения (в проде прикрыты Caddy basic-auth). Fail-open при
  пустом `API_KEY` (`auth.clj:34-37`) — с учётом истории `#env → nil` стоит проверить
  на VPS, что ключ реально резолвится.

### SPA
- **Кнопка Sync мертва везде, кроме Pulse + вечный спиннер.** `events.cljs:713-742`
  матчит старые значения страниц (`:pnl`, `:products`, `[:report t]`), а после
  реструктуризации навигации страницы — `[:finance :pnl]`, `[:products :skus]`,
  `[:treasury :cashflow]`… → кэш сброшен, sync-state `:running`, load-событие не
  диспатчится, `::refresh-finished` не приходит. ПОДТВЕРЖДЕНО.
- **Обязательства: тихий сброс базиса.** После «Погасить»/добавления
  `::treasury-obligation-mutated` (`events.cljs:1227-1233`) перезагружает summary и
  динамику **без mode** → бэкенд-дефолт `:actuals`, а тумблер всё ещё показывает
  «С плановыми» — три виджета одного экрана в разных базисах.
- **Гонки out-of-order ответов на всех загрузчиках** (`events.cljs:212-235` и далее):
  ответ раннего медленного запроса перезаписывает данные позднего — цифры WB под чипом
  «все МП». Нужен guard «ответ соответствует текущим фильтрам».
- **Оси Pulse-графиков захардкожены на май**: `pulse.cljs:42` — подписи `NN.05` для
  любого периода (в июле графики подписаны маем; при >31 точке — «32.05, 33.05…»).

## 3. P2 — средние (кратко)

Расчёты/представление:
- `sku-list :buyout` = sales-базис под тем же ключом, что Pulse отдаёт orders-базис
  (`marker.clj:1407` vs `:782`) — один SKU: 97 % в товарах, 40 % на Pulse.
- `sku-list :margin` — третье определение маржи (`(for-pay−cogs)/max(1,for-pay)`;
  `marker.clj:1404-1406`), кламп даёт −50 000 % при for-pay≈0.
- Два живых варианта max-ДРР расходятся: UE/Pulse включают deduction/additional,
  sku-list — нет (`unit_economics.clj:132`, `marker.clj:922` vs `:1325`).
- `:mp-commission` в Pulse-costs отрицательный и `:source` всегда `:none`
  (`pnl.clj:320` + гейт `pos?` в `marker.clj:283`).
- User-metrics: слаги `:opex`/`:ebitda` из whitelist не резолвятся в ключи строк P&L
  (`report_schemas.clj:100-134`) — метрика валидируется, но вечно рендерит «—»;
  рекурсия валидатора/интерпретатора не ограничена по глубине (StackOverflowError —
  Error, не ловится `catch Exception` → не 422).
- OPEX без дневной пропорции: окно 1–7 июня абсорбирует весь месячный OPEX
  (`opex.clj:141-159`) — по контракту, но искажает недельную management-прибыль.
- Auto-rules: правка суммы правила не обновляет уже материализованные месяцы; после
  ручного удаления авто-строки rematerialize воскрешает её рядом с ручной заменой
  (`opex.clj:212-244`).
- Легаси-водопад `/charts` вычитает `:wb-reward` с подписью «Комиссия МП»
  (`web/api/charts.clj:176-190`) — противоречит канону F-1.
- УСН-база = for_pay и НДС от gross без вычета возвратов — соответствуют LOCKED-контракту
  015, но расходятся с практикой ФНС для агентской схемы (контрактный caveat).

Нормализация:
- Ozon orders: берётся только первый продукт постинга (`ozon/transform.clj:36-55`) →
  недосчёт placed/cancelled для мульти-товарных постингов (sales-путь корректен).
- YM orders: article = первый item, qty/price = вся корзина (`ym/transform.clj:118-133`).
- WB orders/sales ingest без пагинации (`flag 0`, один GET) — тихая обрезка на больших
  объёмах; `materialize-orders!` NPE на nil-дате (`materialize.clj:87-91`).
- `sales`/`orders` PK без marketplace (`db.clj:29-51`) — кросс-МП коллизия id тихо
  перезапишет строку.
- YM finance-даты пишутся без `->iso-datetime` (`ym/transform.clj:365-371`) — сейчас
  чисто, но смена формата апстрима тихо выкинет строки из всех окон.
- `materialize-ozon-services!` небезопасен standalone после respread
  (`materialize.clj:469-488`) — только внутри полного `materialize-finance!`.

API/SPA:
- `Long/parseLong` без try/catch (`treasury.clj:52-70`, `tax_opex.clj:27-36`…) →
  `?page=abc` = 500 вместо 400; `.getMessage` в теле 500 у treasury-хендлеров.
- Мутации v1 — transit-only (JSON-тело тихо валидируется против `{}` → 422); битый
  transit проглатывается и даёт вводящие в заблуждение 422.
- `settle` сверх остатка → 500 вместо 422; obligations-список без `:counterparty-name`
  (контракт его обещает; SPA показывает «Контрагент #N»).
- Мутация операции не обновляет балансы счетов/cashflow на том же экране
  (`events.cljs:1114-1119` — комментарий обещает, кода нет).
- `format-decimal-str` обрезает копейки вместо округления; «−0 ₽» для −0.75
  (`format.cljs:138-153`). ROAS-дельта в UE — произвольный ×10 (`unit.cljs:343`).
- P&L per-SKU Δ% всегда «—» (бэкенд не отдаёт `:delta-pct`, `marker.clj:1282-1291`).
- Pulse forecast-карточка: `:month-plan` захардкожен nil (`marker.clj:946` TODO) —
  планы 017 в Pulse не подключены.
- Пресеты периодов off-by-one: «7 дней» = 8, «30» = 31 (`api.cljs:94-98`).
- Порог «реклама отсутствует» ловит только `:missing`; присутствующий мусор (0.25 ₽)
  проходит как реальные данные → ДРР≈0, ROAS астрономический, без бейджа.

## 5. Покрытие бизнес-задач (сводка coverage-аудита)

8 из 10 спек содержательно доставлены кодом и тестами. Два системных разрыва —
**активационные**, не логические: таймер бота не подключён (P0-4) и у казначейства нет
притока данных (P0-5). Плюс долг живой валидации: 014 FR-006/007, ненулевой ad_cost 011,
015 SC-001, визуальные чек-листы 019 — всё проверено на фикстурах, не на проде.

- **P&L «по копейке»** — покрыт, со звёздочкой WB-рекламы (P0-1) и недоказанной
  live-репликой новых фиксов.
- **Unit-экономика** — покрыта; не хватает effective-dated себестоимости (P2, вне плана).
- **Management P&L (налоги/OPEX)** — покрыт; canonical-formulas.md:228 всё ещё содержит
  противоречащее «tax — out of scope MVP» (015 FR-017 не дочищен).
- **Treasury/ДДС** — механика отличная (decimal-строки, копеечные инварианты, авто-
  классификация), но пустая без ручного ввода (P0-5); пропорция месяцев — мёртвый код
